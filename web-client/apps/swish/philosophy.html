Reasoning with a node-resident program

Consider the following pure Prolog program: 

p(X) :- q(X), r(X).

q(a).
q(b).
q(c).

r(b).
r(c).

... 

?- call(p(X)).

This looks as basic and primitive as it can possibly be, but there are two things going on here: computation and interaction. We can choose to separate them.

reasoning is a form of computation
communication is a form of interaction 
?- spawn(p(X)).

What happened here is that the spawned process performed part of the computation – it computed/reasoned to the first solution (X = b) using SLD resolution (including backtracking on failure) and then terminated. It did not report the result!

Thus it is clear that the process computes but doesn't communicate, and reasons but doesn't interact.

A shell hides this by always displaying the answer to a function application or (as in our case) the call of a goal.

We side here with Peter Wegner and A Turing machine is shutting out the world during the process of computation, whereas with an Interaction machine interaction with the external world -- e.g. with a human agent -- during computation is possible.

Wegner P (1997) Why interaction is more powerful than algorithms. In: Communications of the ACM, v.40 n.5, p.80-91.

Web Prolog is a hybrid of Prolog and Erlang

We have shown that most Prolog programs can be ported to Web Prolog (that's easy) and we have demonstrated that many Erlang programs can be ported as well (even though that's somewhat trickier). Is all well and dandy then? Does it really work? Can Web Prolog be considered a reasonably hybrid between Prolog and Erlang?

It's probably wise to be suspicious of unexpected interactions between language features and possible impedance mismatches between the two paradigms – between Prolog's relational, non-deterministic programming model and Erlang's functional and message passing model. How well do the Erlang-ish constructs mix with Prolog, with the dynamic database and with backtracking for example?

A dynamic database fridge

As hinted at in the introduction, in Web Prolog every actor is equipped with its own private dynamic database. This gives us an alternative way to maintain the state of an actor. Here's how the fridge simulation may be written using assert and retract: 

start(Pid) :-
    spawn(fridge, Pid, [
        src_text("
        
            :- dynamic food/1.
    
            fridge :-
                receive({
                    store(From, Food) ->
                        assert(food(Food)),
                        self(Self),
                        From ! ok(Self),
                        fridge;
                    take(From, Food) ->
                        self(Self),
                        (   retract(food(Food))
                        ->  From ! ok(Self, Food),
                            fridge
                        ;   From ! not_found(Self),
                            fridge
                        );
                    terminate ->
                        true
                }).    
        ")
    ]).

Let's try this by running the following goal ... 

?- start(Pid),
   self(Self),
   Pid ! store(Self, meat),
   Pid ! store(Self, cheese),
   Pid ! take(Self, meat),
   Pid ! take(Self, cheese),
   Pid ! take(Self, cheese).
   Pid ! terminate.

... and then check the content of the top-level mailbox: 

?- flush.

The dynamic database is a non-logical extension to Prolog and the use of it for managing state is usually frowned upon by Prolog programmers since changes to the database are not reverted on backtracking. Also, clauses of dynamic predicates are like destructible global variables and therefore subject to all the common problems with such objects. (See here for good advice when dealing with the dynamic database in SWI-Prolog.) However, for storing really, really large quantities of food in our fridge this might still be the best solution.

A Prolog-style dynamic database is not something that Erlang has, and we need to determine if it could lead to problems when mixed with Web Prolog. The answer is "no" since Erlang does in fact have something which plays the same role as the dynamic database.

In Erlang, each process has a local store called the process dictionary and Erlang support a handful of built-in functions for manipulating it. Process dictionaries are private and cannot be modified by any other process. Erlang programmers are advised again using it since it destroys referential transparency and makes debugging difficult. (See here for good advice when dealing with the process dictionary in Erlang.)

Perhaps the right conclusion to draw here is that the dynamic database in Web Prolog is not more dangerous than the process dictionary in Erlang. It is arguably also more powerful and useful than the process dictionary.

The question if Web Prolog should allow for some kind of memory over and above what can be passed around in the arguments of predicates or asserted in the private dynamic database is interesting. Here, Erlang supports a built-in term storage known as ETS and the distributed database Mnesia. In line with the focus of Web Prolog on the web domain, a solution based on RDF might be appropriate and easily implemented in SWI-Prolog.

Controlling backtracking

How well do the Erlang-ish messaging constructs combine with backtracking? Let's turn to a tiny but illustrative example which seems to show that the mix is both sound and easy to understand.

Suppose the goal given in the first argument to spawn/2 has more than one answer, a query such as ?-ancestor_descendant(mike, Who) for example. Below, we call the goal, send the result back to the calling process, and then use receive/1 to listen for a message of the form next or stop before terminating. 

?- self(_Self),
   spawn(( ancestor_descendant(mike, _Who),
           _Self ! _Who,
           receive({
               next -> fail;
               stop -> 
                   _Self ! stopped
           })
        ), Pid).

The first answer to the query should now be present in the mailbox of the top-level pengine and can be inspected if we call flush/0:

?- flush.

At this point, the receive call is blocking while waiting for messages to show up in the mailbox. To check if there are more solutions to our query we send a message next to the actor. This will cause the receive to fail and thus force the backtracking that will trigger the search for the next solution: 

?- $Pid ! next.

Did we get more solutions?: 

?- flush.

Below we show how to stop asking for more solutions to the query. This time we wait for the expected stop message using receive instead of flushing the mailbox: 

?- $Pid ! stop,
   receive({Message -> true}).

This suggests that Prolog's backtracking mechanism is perfectly compatible with and also complements the proposed Erlang-like mechanism for spawning actors and handling the communication between them. We need to observe, however, that the goal to be solved in the above example is hard-coded into the program, and that the program handles neither failure of the spawned goal, nor exceptions thrown by it.

A generic encapsulated search procedure

In this section we define a generic encapsulated search predicate by using a meta-predicate call_cleanup/2 and by specifying a small set of custom messages carrying answers that needs to be returned to the calling process. Note that call_cleanup/2 is here used not only to call a goal, but also to check if any choice points remain after the goal has been called or backtracked into. 

search(Query, Pid) :-
    self(Self),
    spawn(query(Query, Pid, Self), Pid, [
        monitor(true),
        src_predicates([query/3])
    ]).
    
query(Query, Self, Parent) :-
    call_cleanup(Query, Det=true),
    (   var(Det)
    ->  Parent ! success(Self, Query, true),
        receive({
            next -> fail;
            stop -> true
        })
    ;   Parent ! success(Self, Query, false)
    ).

Here's how to call search/2 with a query: 

?- search(member(_X,[a,b,c]), Pid).

Let's flush to inspect the first solution:

?- flush.

We then ask for the next solution:

?- $Pid ! next,
   receive({Message -> true}).

Tell the actor to stop:

?- $Pid ! stop,
   receive({Message -> true}).

What about failures and exceptions? Well, since we used a monitor the information provided in the down message tells us what we need to know. Try the following queries and see for yourself: 

?- search(false, _Pid),
   receive({Message -> true}).


?- search(not_defined, _Pid),
   receive({Message -> true}).

This section offered some insight into how encapsulated search works when understood as an actor that adheres to a particular protocol. Seemed tricky? Here's the good news: you don't need to build your own implementations of encapsulated search, but can rely on the built-in pengine abstraction – the special kind of actor introduced in the beginning of the tutorial.

The pengine behaviour

A pengine is like a first-class interactive Prolog top-level, accessible from Web Prolog as well as from other programming languages. For example, recall that the shell running in the area to the right is an application written in HTML, CSS and JavaScript that is talking to a pengine – your pengine. In this section we'll show how to spawn other pengines, and how to use your shell to talk to them via your top-level pengine.

Whereas encapsulated search (as implemented above) allows only one query per process created, a pengine optionally allows a full session per process, possibly involving many queries. This is why we sometimes refer to a pengine as an encapsulated Prolog session.

As we stated in the introduction, a pengine is also an actor, a special kind of actor. Its behaviour differs from the behaviours of other kinds of actors in that it adheres to a protocol specific to the communication between a client and a pengine – the Pengine Communication Protocol (PCP).

The statechart below depicts the PCP. The client can be any kind of process (including another pengine) capable of sending the type of messages and signals in bold to the pengine. The pengine is responsible for returning the type of messages with a leading / back to the client.

Web Prolog comes with a set of built-in predicates that allows a client to spawn a local or remote pengine (pengine_spawn/1-2), and to send the type of messages in bold to it (pengine_ask/2-3, pengine_next/1-2, pengine_stop/1, pengine_respond/2, pengine_abort/1 and pengine_exit/1). Under the hood, those predicates are using !/2 and receive/1-2 to do their thing so they are mostly there for convenience and for shielding the programmer from the raw details of the protocol. Here's how to spawn a local pengine: 

?- pengine_spawn(Pid, [exit(false)]).

The value of the exit option is either true (default) or false). If false, it tells the pengine to run a full session, i.e. to not terminate once the first query that we are going to ask it has run to completion. In other words, this is an option that allows us to switch from running a one-shot encapsulated search task into running a full encapsulated Prolog session.

The pengine is now initialised and the protocol is in state s1. Here's how to ask the pengine a query: 

?- pengine_ask($Pid, member(_X,[a,b,c])),
   receive({Message -> true}).

This took the protocol from state s1 to state s2 and then to state s4. A success message was sent back to the client during the transition from s2 to s4. The reason we end up in s4 rather than in s1 is that the message (which was "produced" inside s2) has true in its third argument, indicating that more solutions may exist.

Here's how to use pengine_next/1 to request the next solution to our query: 

?- pengine_next($Pid),
   receive({Message -> true}).

The protocol transitioned first to state s2, and then back to s4 again.

When we don't need more answers, we call pengine_stop/1: 

?- pengine_stop($Pid),
   receive({Message -> true}).

We are back in state s1 and the pengine is ready to accept new queries. Now, let's try this: 

?- pengine_ask($Pid, pengine_output(p(a))),
   receive({Message1 -> true}),
   receive({Message2 -> true}).

This took the protocol to state s2 and then over the transition labeled /output and back to state s2 and finally to s1 again. Two messages were sent back to the client: one message representing the output produced by the call to pengine_output/1, and one message representing the success of the call itself.

Next, let's try this: 

?- pengine_ask($Pid, pengine_input(myprompt, _Answer)),
   receive({Message -> true}).

This took the protocol to state s2 and then directly to state s3 while it sent back a prompt term. It now waits in state s3.

This is how we can respond to the prompt: 

?- pengine_respond($Pid, hello),
   receive({Message -> true}).

This took the protocol to state s2 and then directly to state s1 while sending back a success term to the client.

Having spawned a process running a goal, it sometimes happens that it loops indefinitely, so that we may need to terminate it by force.

To see how, let's ask a query that takes a while to terminate ...

?- pengine_ask($Pid, forall(between(1,30, _N), (io:writeln(_N), sleep(1)))).

... just to be able to show how to abort it:

?- pengine_abort($Pid),
   receive({Message -> true}).

BUG! The abort generated the wrong response. Should have been abort(<pid>).

Note that aborting the running query doesn't kill the pengine. It's still alive, ready to accept new queries. To kill a pengine from the outside, you'd need to call pengine_exit/2 (or just exit/2). To do it from the inside, i.e. to make it commit suicide, you would need to use pengine_exit/1 (or just exit/1). More on this later.

Pengines deferring messages

One consequence of the message-deferring behaviour of receive/1-2 described earlier is that messages, to some extent, are allowed to arrive to the mailbox of a pengine in the "wrong" order. Let's look at an example where we first spawn a new pengine: 

?- pengine_spawn(Pid).

Now, let's make a deliberate "mistake" and call pengine_next/1 to request the next solution to a query that hasn't yet been asked:

?- pengine_next($Pid).

We use flush/0 to inspect the result:

?- flush.

Right, the mailbox was empty since the protocol obviously wasn't in a state where it could react on the next message. The message was therefore deferred and it's not until pengine_ask/3 is called and the protocol changes states that the message next can have an effect. Let's now ask the pengine a query ... 

?- pengine_ask($Pid, member(_X,[a,b,c])).

... and use flush/0 to inspect the result:

?- flush.

Yes, that's right – now two messages appeared in the mailbox!

Pengines and options

A pengine is a programming abstraction built on top of an actor and as a consequence pengine_spawn/2 inherits the options that spawn/3 offers. Here's how to use some of them when creating a monitored session running remotely: 

?- pengine_spawn(Pid, [
       exit(false),
       monitor(true),
       node('{{host}}'),
       src_list([p(a),p(b),p(c),p(d),p(e),p(f),p(g)])
   ]).

The predicate pengine_ask/3 introduces two new options, template and limit. Here's how they may be used when querying the pengine just created: 

?- pengine_ask($Pid, p(_X), [
       template(_X),
       limit(2)
   ]),
   receive({Message -> true}).

The predicate pengine_next/2 is used for retrieving more solutions, and will get two of them this time too, just because the previous call to pengine_ask/3 set this limit: 

?- pengine_next($Pid),
   receive({Message -> true}).

But pengine_next/2 supports the option limit as well, so that we are allowed to increase (or decrease) the number of solutions that we would like to get: 

?- pengine_next($Pid, [
       limit(4)
   ]),
   receive({Message -> true}).

At some point, we may want to terminate the session. This is where the exit/2 predicate comes in. Since our pengine is monitored we can expect a down message to appear in our top-level mailbox: 

?- exit($Pid, please_die),
   receive({Message -> true}).

Using the Web APIs

A Web Prolog runtime system – or what we have referred to as a node – is equipped with a small but comprehensive set of web APIs, using both WebSocket and HTTP as transport protocols, over which a client can run Web Prolog programs defined by the owner of the client, the owner of the node, or a combination of both. In this section we will take a closer look at these web APIs and also demonstrate how they can be used.

The web APIs are not an afterthought. They are extremely important. Indeed, if we did not use HTTP or WebSocket, we would not be on the Web. We use both, and the whole distribution layer of Web Prolog depends on them, for openness, as well as for security. The web APIs must be capable of everything that the predicate APIs are capable of, and vice versa. Otherwise they would not be able to support the distributed programming model of Web Prolog.

Using the WebSocket API

A Web Prolog node supports an asynchronous and stateful web API that relies on the WebSocket transport. Offering an almost total control over a pengine it currently serves the shell (which despite its apparent simplicity is a fairly demanding web application) as well as the predicate API with its node option to spawn/2-3and its network transparent send primitive !/2.

For an example of how the WebSocket API can be used, you need to look no further than the source code implementing the shell. It's a bit too messy for a tutorial though. (The implementation relies a lot on a library called jQuery Terminal. Without that, it would have been a lot messier.)

We have prepared two much simpler (and self-contained but useless) applications that you can try by selecting the links below. Upon page load, the first application will create a WebSocket connection to the node, spawn a pengine there, and (when having received a pid) ask the query ancestor_descendant(mike, Who). For demonstration purposes, the limit option will be set to 2. Here's the complete HTML and JavaScript source code for this application: 

<!DOCTYPE html>
<html lang="en">
  <head>
    <meta charset="utf-8"/>
    <title>Example 1</title>
    <script>
      var ws = new WebSocket('ws://localhost:3060/ws','pcp-0.2');
      ws.onopen = function (message) {
        ws.send(JSON.stringify({
          command: 'pengine_spawn'
        }));
      };
      ws.onmessage = function (message) {
        var event = JSON.parse(message.data);
        if (event.type == 'spawned') {
          ws.send(JSON.stringify({
            command: 'pengine_ask',
            pid: event.pid,
            query: 'ancestor_descendant(mike, Who)',
            options: '[limit(2)]'
          }));
        } else if (event.type == 'success') {
          document.getElementById("output").innerHTML += 
              JSON.stringify(event.data) + "<br/>";
          if (event.more) {
            ws.send(JSON.stringify({
              command: 'pengine_next',
              pid: event.pid
            }));
          }
        }
      };   
    </script>
  </head>
  <body>
    <div id="output"></div>
  </body>
</html>

Using the HTTP ask API

In addition to the asynchronous and stateful API described above, a Web Prolog node offers a synchronous and stateless web API using HTTP as a transport. Alternatively, although this isn't a term we like to use since it's so vague, this API can be described as RESTful. In any case, statelessness is definitely part of being RESTful.

To be considered stateless, a node should be able to (as it were) "forget" everything about the previous request as soon as it has returned a response to a query, and the next request to the node should be formed as if the previous one never occurred. It means, for example, that uses of HTTP cookies and other ways to manage sessions are ruled out.

In addition to a mandatory query parameter, the API also supports template and limit with semantics identical to the corresponding predicate API options. Moreover, the API supports an offset parameter which is of great importance. Using a combination of offset and limit, a client can ask for a "slice" of solutions starting at the offset indicated, and with a length no greater than the value of the limit parameter. Finally, the format parameter determines the format of messages returned to the client. Currently prolog, json (default) and json-s are recognised values.

Let's look at an example. When the link below is selected, a request to find a descendant of Mike is made:

 {{host}}/ask?query=ancestor_descendant(mike,Who)

But that's just one of his descendants and we already know that they are three. We can retrieve the remaining answers by making the following request, i.e. with offset set to 1 this time, and with limit set to 2:

 {{host}}/ask?query=ancestor_descendant(mike,Who)&offset=1&limit=2

The challenge here has been to make this efficient when queries have more than one solution (and in particular if there are very many), i.e in cases when backtracking over HTTP is called for. Using our approach, it's likely (but not guaranteed) that the work that generated the first solution doesn't have to be repeated. Pengines are involved here too, but they are anonymous. An anonymous pengine running on the node serves as a form of Prolog state cache. The cached state can only be used once though, and there's a possibility that another client made use of it before you had a chance to do so. If so, no real harm is done.

This API doesn't support I/O, and is therefore only suitable for applications in need of a "logic server". Nor does it support uses of assert and retract or is capable of supporting sessions.

Using the HTTP send API

By means of the HTTP send API, and given a pid or a registered name, you are able to send a message to an actor with that pid or name. To demonstrate how this works, we ask you to once again subscribe to the publish-subscribe service: 

?- self(Self),
   pubsub_service ! subscribe(Self),
   repeat,
   io:write("Waiting for a message ..."),
   receive({
       msg(Message) ->
           io:format("Received: ~p", [Message]),
           fail
   }).

Here's a link making an API request that will publish the term hello to the service which will then forward it to all subscribers:

 {{host}}/send?name=pubsub_service&message=publish(hello)

You should now have received the message "hello". Such a request can be made by any client that knows the name of the service.

High-level remote procedure calls

We have already demonstrated how to create remote pengines or other actors using the node option of pengine_spawn/1-2 and spawn/2-3. In both cases communication with the remote actor is asynchronous and uses the WebSocket transport. Web Prolog also supports two kinds of more high-level predicate APIs for making remote procedure calls, one which is synchronous and very easy to use, and one which is asynchronous and somewhat harder to use. In both cases and in contrast to pengine_spawn/1-2 and spawn/2-3 they don't support sessions, only one-shot queries.

Non-deterministic RPC

Web Prolog offers a predicate rpc/2-3 capable of non-deterministic remote procedure calls. Here's our first example of its use: 

?- rpc('{{host}}', member(X,[a,b,c])).

By default, rpc/2-3 uses the RESTful HTTP API to communicate with the remote node.

The following example shows that thanks to the Prolog state cache the first (slow) solution isn't recomputed when we ask for the second one, despite the fact that rpc/2 makes two totally separate HTTP requests: 

?- rpc('{{host}}', (sleep(1),X=a ; X=b)).

For the purpose of reducing the number of network roundtrips, the option limit (inherited from pengine_ask/3) may be passed: 

?- rpc('{{host}}', between(1, 12, N), [
       limit(5)
   ]).

What this example tries to convey is that although the call only makes three network roundtrips in order to generate all twelve solutions, the behaviour of the call, as seen from the point of view of the client, doesn't change. Indeed, the limit option serves as a performance optimisation only, and has nothing to do with logic and the declarative reading of the query.

The transport option allows us to use the WebSocket transport instead of HTTP. In order to inject code into the process, the option src_text (inherited from pengine_spawn/2 which in turn inherited it from spawn/3) may be passed: 

?- rpc('{{host}}', p(X), [
       transport(ws),
       src_text('p(a). p(b). p(c).')
   ]).

Bringing data to the code

The special purpose URI localnode serves as the default value of the node option for both spawn/3 and pengine_spawn/2. Thus it follows, perhaps a bit counter-intuitively, that rpc/3 in combination with the src_uri option can also be used to "bring data to the code": 

?- rpc(localnode, p(X), [
       src_uri('https://gist.githubusercontent.com/torbjornlager/21583e7238dc855f94f6b5f2f48788bb/raw/p.pl')
   ]).

Promises (asynchronous RPC)

A promise is a type of RPC method that doesn't suspend the caller until the result is computed. Instead, a reference is immediately returned, which can later be used by yield/2-3 to collect the answer. The reference can be viewed as a promise to deliver the answer. Let's try this: 

?- promise('{{host}}', member(_X,[a,b,c,d]), Reference).

The predicate yield/2-3 returns the promised answer from a previous call to promise/3-4. If the answer is available, it's returned immediately. Otherwise, the calling process is suspended until the answer arrives from the node that was called. Note that this predicate must be called by the same process from which the previous call to promise/3-4 was made.

?- yield($Reference, Answer, [timeout(4)]).

To get a glimpse of the behaviour when no answer is available, you may want to run the query again ...

The above use of promise/3-4 and yield/3 only gave the first answer to the query. To produce other answers, we can pass the option offset. The limit option is also available, allowing us to ask for more than one answer at a time:

?- promise('{{host}}', member(_X,[a,b,c,d]), Reference, [
       template(_X),
       offset(1),
       limit(2)
   ]).

Let's call yield/3 again:

?- yield($Reference, Answer, [timeout(0.5)]).

Note that if we would now request the remaining answers by re-running the call to promise/4 with offset set to 3 this time, it's likely (but not guaranteed) that the work that generated the three first solutions does not have to be repeated. Again, this is due to the RESTfulness of the HTTP API, where an anonymous pengine running on the node serves as a form of cache.

Note that promise/4 operates over HTTP only, which is useful if a WebSocket connection is not available.

An alternative IDE. Want to try a more sophisticated online Prolog IDE? Head over to SWISH, which is a mature web-based IDE to Prolog, featuring semantic source code colouring, Prolog notebooks, tools for collaboration, and other niceties.

In September 2017, SWISH had been online for three years. It is extensively being used for Prolog education. At the time of writing this, the main server now has 35,420 programs stored with 68,806 revisions. Usage is strongly correlated with academic holidays, ranging from 36,415 queries per week (July 31 - August 6, 2017) to 125,946 (May 7-14, 2017). Google link search (\verb$link:swish.swi-prolog.org$) reports 8,490 links. The popularity of the Github repository, 198 stars and 48 forks compared to the main SWI-Prolog repository with 223 stars and 53 forks is another indication of the impact. (To be compared with Erlang's 6,374 stars and 1,708 forks.)

Please understand, though, that the present version of SWISH doesn't implement Web Prolog, but something slightly different. In the future, we plan to integrate Web Prolog with SWISH.
